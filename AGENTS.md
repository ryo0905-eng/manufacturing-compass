# AGENTS.md

このドキュメントは、Manufacturing Compass を開発する AI 開発エージェント向けの最重要ルールです。

## ミッション

Manufacturing Compass は、半導体業界への転職を、もっとわかりやすくするためのキャリアプラットフォームです。

今すぐ転職できる人だけでなく、今はまだ届かない会社にも、いつか挑戦できる道筋を示します。ユーザーが「次に何をすればよいか」まで理解できるサービスにしてください。

## 開発方針

- SEO とコンテンツ構造を優先する
- 過剰設計しない
- MVP では半導体業界に特化する
- 電子部品、FA、ロボット、自動車、医療機器は将来拡張とする
- ユーザーの意思決定に役立つ情報を優先する
- アフィリエイトよりも信頼性を優先する

## 実装優先順位

1. Career Compass によるアプリ的な現在地チェック
2. 相談論点が明確になる自然な転職エージェント CTA
3. SEO に強い情報設計
4. 半導体業界地図
5. 企業データベース
6. 企業詳細ページ
7. 企業比較ページ
8. 企業ページ内のキャリア準備情報

## MVP で作るもの

- トップページ
- Career Compass
- 半導体業界地図
- 企業一覧
- 企業詳細ページ
- 企業比較ページ
- 基本的な検索・フィルター
- 企業ページ内のキャリア準備情報
- 転職エージェント CTA
- SEO メタデータ
- sitemap

Phase 1 の MVP は、静的データで小さく公開します。最初から Supabase、認証、ユーザー保存、AI 診断に依存しないでください。

Career Compass は、ログインや個人情報保存を行わない静的ルールベースの体験として実装します。これはユーザーを判定するためではなく、相談論点と次の行動を整理するための中核導線です。

## MVP で作らないもの

- ログイン
- 口コミ
- AI チャット
- コミュニティ機能
- 個人情報保存型のキャリア診断フォーム
- AI キャリア診断
- 求人応募機能
- 求人データ連携
- ユーザーアカウント
- 電子部品、FA、ロボット、自動車、医療機器領域

## コーディング方針

- 将来 Next.js で実装する際は App Router を使う
- Server Components を基本にする
- Client Components はインタラクションが必要な箇所に限定する
- TypeScript を厳格に使う
- シンプルで読みやすいコードを優先する
- データ構造を明確にし、表示ロジックとデータ定義を分離する
- まず小さく公開し、需要検証しながら改善する
- 初期データは `src/data` などのローカル静的データで管理し、Supabase 移行を前提に型を崩さない

## Next.js 作業時の検証・実行ルール

Codex 環境では、Next.js 16 の Turbopack と sandbox の相性により、`next dev` や Turbopack build がプロセス作成・ポート bind で失敗することがあります。

このプロジェクトでは根本対策として、`package.json` の `build` は `next build --webpack` を使います。Next.js 16 は Turbopack がデフォルトですが、Codex ローカル検証では Webpack build を標準にしてください。

ただし、Codex 環境では Webpack build でも最適化フェーズでログが進まないことがあります。日常のコード確認は `npm run typecheck` を第一確認にし、フルビルドは必要な時だけ実行してください。

- 原則として、Codex 内で dev サーバーを長時間起動しっぱなしにしない
- 画面確認が必要な時だけ `npm run dev` を起動し、確認後は必ず停止する
- dev サーバーを起動したまま `npm run build` や `npm run typecheck` を実行しない
- build 前に動作が不安定な場合は、生成物である `.next` を削除してから実行する
- 日常確認はまず `npm run typecheck` を使う
- `npm run build` は Webpack 指定になっている前提で、フルビルド確認が必要な時だけ使う
- `next build` を直接実行しない。直接実行すると Turbopack が使われ、sandbox のポート制限に当たりやすい
- `npm run build` が sandbox 由来の権限エラーやポート制限で失敗した場合のみ、同じコマンドを権限昇格して一度だけ再実行する
- 権限昇格後も Next CLI がログを出さずに止まる場合は、何度も再試行しない
- その場合は変更差分を小さく保ち、静的確認、対象ファイル確認、Vercel のビルドログで確認する
- Vercel 連携後の最終確認は、ローカルで無理に粘るより Vercel のビルド結果を優先する
- `tsconfig.json` に Next が `.next/types/**/*.ts` を追加することがあるため、単体 `tsc` が重い場合は原因を切り分ける

推奨する基本手順:

```bash
# dev サーバーを使っている場合は先に停止する
# 必要な場合だけ生成キャッシュを削除する
rm -rf .next

# まず型チェックで確認する
npm run typecheck

# フルビルド確認が必要な場合だけ Webpack 指定の本番ビルドを確認する
npm run build
```

Codex への作業指示では、必要がない限り「dev サーバーは起動しなくてよい。修正後は `npm run typecheck` を確認する。フルビルドが必要な場合のみ `npm run build` を一度だけ確認する。`next build` 直実行は避ける」と扱ってください。

## UI 方針

- 清潔感、余白、信頼感を重視する
- プロフェッショナルで少し Apple 風の印象を目指す
- 派手な装飾よりも読みやすさを優先する
- 怪しいアフィリエイトサイトに見えないようにする
- スマホで読みやすいことを最優先にする
- キャリア現在地 UI は、判定される印象ではなく、前向きに背中を押す印象にする
- Phase 1 では診断 UI ではなく、企業ページや比較ページ内の「準備ポイント」として表現する
- Career Compass は例外として、個人情報を保存しない静的ルールベースの現在地チェックとして扱う

## SEO 方針

- ページごとに検索意図を明確にする
- 企業名、職種、業界用語、比較キーワードを意識する
- 内部リンクで業界地図、企業詳細、比較、記事、キャリア準備ページをつなぐ
- 構造化データ、sitemap、メタデータを設計段階から考慮する
- コンテンツを継続的に追加し、SEO 資産を積み上げる

## コンテンツ方針

- 公開情報を使う
- 企業公式サイト、IR、統合報告書、求人情報、信頼できる情報源を優先する
- AI で下書きしてもよいが、人間による編集・確認を必須にする
- 年収、RSU、福利厚生などを推測で書かない
- 不確かな情報は不確かと明記する
- 情報ソース URL と最終更新日を必ず持つ
- 難しい半導体用語はわかりやすい日本語で説明する

## データ正確性の方針

- 企業情報、財務情報、採用情報には情報ソース URL を付ける
- 最終更新日を保持する
- 古い情報は明示する
- 推測と事実を混ぜない
- 重要な数値は一次情報または信頼できる二次情報で確認する

## アフィリエイト倫理

- アフィリエイトリンクは自然に配置する
- ユーザーに強く売り込まない
- CTA は有益な情報提供の後に出す
- サイトの信頼性を最優先する
- 収益化は、ユーザーの業界理解・キャリア理解を助けた結果として発生させる

## 将来拡張方針

- MVP 後に個人情報保存やAIを伴う本格的なキャリア診断を追加する
- AI キャリア診断は十分なコンテンツとデータが揃ってから検討する
- 半導体領域で検証した後、電子部品、FA、ロボット、自動車、医療機器へ拡張する
- 求人連携やユーザーアカウントは、需要検証後に追加する

## 長期的なキャリア支援の思想

ユーザーを不必要に落ち込ませない表現を使います。

避ける表現:

- この会社は無理です
- あなたには難しいです
- 経歴が足りません

望ましい表現:

- 今すぐはチャレンジングです
- この経験を積むと可能性が上がります
- まずは近い職種・企業から狙うのも現実的です
- 半年後、1年後に向けて準備できます

今狙える会社と将来のストレッチ目標を区別し、足りないスキル、経験、英語力、資格、実績を前向きに整理してください。
