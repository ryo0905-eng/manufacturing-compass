# Article Workflow

## 目的

RYO の実体験を一次情報として、AI をインタビュアー・編集者・調査補助として使う。AI が体験、数字、判断を推測して記事を作ることは禁止する。

## 情報の保管場所

- 非公開の取材メモ、音声、勤務先や顧客を推測できる情報: 非公開の Notion または Google Drive
- 公開を承認した記事本文、公開情報の出典、更新履歴: Git リポジトリ
- Search Console の表示回数、クリック、順位、次回更新日: 非公開のコンテンツ管理表

生の取材メモや公開不可情報は Git リポジトリへ保存しない。

## 記事ステータス

1. アイデア
2. 取材待ち
3. 構成確認
4. 初稿
5. 事実確認
6. 公開準備
7. 公開済み
8. 改善対象

## 制作フロー

### 1. テーマ選定

- Search Console の検索クエリ、表示回数、平均掲載順位を確認する
- 読者の悩みを一つに絞る
- 記事から診断へ進む理由を決める
- 既存記事と検索意図が重複していないか確認する

### 2. 取材

AI は `article-interview-template.md` を使い、一度に質問を並べず、一問ずつ聞く。RYO は箇条書きまたは音声入力で回答してよい。

### 3. 取材メモの分類

- 実体験: RYO が直接経験した事実
- 公開情報: 公式サイト、IR、採用情報、業界団体などで確認した事実
- 一般的な考え方: 実体験と公開情報から整理した助言
- 要確認: 根拠、数字、公開可否が確認できていない内容
- 公開不可: 記事へ使用しない情報

### 4. 構成確認

AI は本文を書く前に、検索意図、結論、見出し、使用する実体験、必要な出典、CTA を提示する。RYO の承認後に初稿を作る。

### 5. 初稿作成

- 実体験を一般論として広げすぎない
- 経験していない職種は、公開求人や公式情報に基づく説明と明記する
- 未確認の内容は `[RYO確認]` とし、推測で埋めない
- 年収、効果額、改善率には対象期間と範囲を付ける
- 記事末尾に今日やることと診断への導線を置く

### 6. 公開前確認

`article-review-checklist.md` を使って RYO が確認する。承認後、AI が `src/content/guides/` へ実装し、テスト・ビルド・プッシュを行う。

### 7. 公開後の確認

- 14日: インデックス、表示クエリを確認
- 30日: 表示回数、クリック、診断開始を確認
- 90日: 追記、統合、タイトル変更の要否を判断

短期の順位変動だけを理由に記事を頻繁に書き換えない。

## コンテンツ管理表の項目

- 記事ID
- 仮タイトル
- 対象検索クエリ
- 検索意図
- 読者の悩み
- ステータス
- 優先度
- 取材メモURL
- 実体験確認状況
- 出典確認状況
- 公開不可情報の確認状況
- 公開URL
- 公開日
- 最終更新日
- Search Console 表示回数
- Search Console クリック数
- 平均掲載順位
- 診断開始数
- 次回確認日
