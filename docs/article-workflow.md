# Article Workflow

最終更新日: 2026-07-19

## 目的

記事を「実体験記事（EXP）」と「公開情報記事（REF）」に分け、事実、本人の受け止め、編集上の提案を混ぜずに公開します。AI は取材、構成整理、調査補助、実装を支援できますが、体験、数値、評価理由を推測して作りません。

## 情報の保管場所

- 非公開の取材・調査メモ: `.private/article-notes/`
- 非公開の下書き: `.private/article-drafts/`
- 記事ID、状態、次回確認日: `.private/article-index.md`
- 公開を承認した本文と出典: `src/content/guides/`
- 実体験記事の機密・承認手順: `.private/article-workflow.md`
- 公開禁止情報: `.private/content-redaction-dictionary.md`
- RYO の文体: `.private/ryo-editorial-voice.md`

`.private/` は Git 管理外です。個人情報、生の取材回答、公開不可情報を公開コードや公開文書へ直接コピーしません。

## 記事IDと種類

### EXP: 実体験記事

転職、職務経歴書、英語、エージェント利用など、RYO の経験・観察・判断が価値の中心になる記事です。

- 本文より先に取材する
- 一般論だけで初稿を作らない
- `.private/article-workflow.md` を必須とする
- RYO が事実、匿名化、公開可否を確認するまで公開しない

### REF: 公開情報記事

工程、装置、材料、企業分類、ランキングなど、公開一次情報の整理が価値の中心になる記事です。

- 実体験を必須にしない
- 官公庁、企業公式、研究機関、業界団体などを優先する
- 非公開の業務情報や顧客固有情報を使わない
- 技術条件、対象時点、企業・方式間の境界を確認する

## 共通ステータス

記事管理では次の語を使います。

1. アイデア
2. 取材待ち / 調査待ち
3. 構成確認
4. 初稿
5. 事実確認
6. 公開準備
7. 公開実装済み
8. 公開済み
9. 改善対象
10. 見送り

`公開実装済み` はコードへ反映済みだが、本番表示やインデックスを未確認の状態です。`公開済み` は本番表示を確認した状態です。見送り記事のIDは経緯を残すため再利用しません。

## 共通フロー

### 1. テーマと境界を決める

- `.private/article-index.md` と既存記事を確認する
- 誰の、どの疑問に答えるかを一つに絞る
- 中心検索クエリと、扱わない隣接テーマを決める
- 記事種別を EXP / REF のどちらかにする
- 読後の自然な次の行動を仮置きする。CTA が不要なら置かない

### 2. 非公開メモを作る

- 記事IDとファイル名を一致させる
- 想定読者、検索意図、状態、作成日を記録する
- EXP は取材メモ、REF は一次情報と使う目的を記録する
- 公開不可・要確認を本文候補と分ける

### 3. 取材または調査を行う

EXP は [`article-interview-template.md`](./article-interview-template.md) と `.private/article-workflow.md` に従い、原則一つずつ質問します。

REF は、主張ごとに一次情報を確認します。URLだけでなく、発行元、確認日、記事内で支える内容をメモします。市場順位や変動数値は対象時点と算出条件を確認します。

### 4. 構成を確認する

本文の前に、次を整理します。

- 読者の疑問と記事の答え
- 見出しと、各見出しが担う範囲
- EXP で使う実体験、または REF で使う一次情報
- 不確かな点と公開しない点
- 必要な図、表、対応関係
- 関連記事と CTA

EXP は RYO の回答の流れと確信の強さを保ちます。REF はハブ記事と隣接記事の役割を奪わないようにします。

### 5. 初稿を作る

- 事実、実体験、受け止め、編集提案を読者が区別できるようにする
- 未確認の内容を推測で埋めない
- 数字に対象期間、範囲、基準時点を付ける
- 記事固有の関係を理解しやすくする時だけ図表を使う
- 同じ説明を本文、表、カード、結論で繰り返さない
- 検索キーワードを不自然に反復しない
- CTA は本文の答えより目立たせない

### 6. 公開前に確認する

- [`article-review-checklist.md`](./article-review-checklist.md) を記事種別に応じて使う
- `.private/content-redaction-dictionary.md` を、本文、メタデータ、引用、構造化データまで照合する
- EXP は `.private/ryo-editorial-voice.md` で本人らしさを確認する
- RYO または指定された人間が、技術内容または実体験、匿名化、公開可否を確認する
- 状態、公開日、更新日、著者、確認者、出典、内部リンクを設定する

### 7. 実装と検証を行う

- `src/content/guides/` へ反映する
- 変更ファイルに限定した静的確認と差分確認を行う
- コード変更で必要な場合だけ `npm run typecheck` を最大1回実行する
- lint は必要な場合だけ最大1回実行する
- build はユーザーが明示した場合だけ実行する
- ブラウザ確認はインタラクティブUIを変更し、確認を求められた場合だけ行う
- commit と push はユーザーが明示した場合だけ行う

検証の正本は `AGENTS.md` です。記事チェックリストが build やブラウザ確認を一律に要求することはありません。

### 8. 公開後に管理表を更新する

- 本番表示を確認したら `公開済み` へ更新する
- `.private/article-index.md` に公開URL、公開日、最終更新日、次回確認日を記録する
- 原則14日後にインデックスと表示クエリを確認する
- 30日以降に表示、クリック、関連導線を確認する
- 変動情報は内容に応じた更新周期を設定する
- 短期の順位変動だけで頻繁に書き換えない

`TASKS.md` は横断プロジェクトの進捗だけを更新し、記事ごとの細かな次回確認は `.private/article-index.md` に集約します。
